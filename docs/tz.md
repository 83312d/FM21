## 1. Обзор проекта

FM21 — автономный сервис интернет-радио с непрерывной аудиотрансляцией (музыка + IT-новости + голосовые объявления). Управление эфиром осуществляется через Telegram-бота. Авторизация слушателей не требуется — открытый доступ.

**Главное правило — геотаргетинг:**

- При запуске плеера определяется тег города слушателя (`city_tag`).
- Объявления и заказы музыки воспроизводятся только для слушателей соответствующего города.
- Специальный тег `all` — глобальный эфир для всех слушателей.

**Целевое поведение:** пользователь открывает сайт → слышит музыку из плейлиста для своего города → каждые 15 минут звуковая перебивка «Сейчас новости» (3–5 сек) → короткая новость (1–2 мин) → снова музыка. Голосовое объявление из бота для этого города встаёт в очередь после текущего трека. Заказ песни через бота попадает в очередь всех слушателей города.

### 1.1. Ключевые принципы

- Непрерывное воспроизведение с плавными переходами (Web Audio API / AudioContext).
- Работа в фоновой вкладке браузера (Media Session API, Page Visibility API).
- Отдельная очередь эфира на каждый `city_tag`.
- Приоритет очереди: **объявления** > **новости** > **заказанная музыка** > **обычная музыка**.
- Ни один блок не прерывает текущий звук — новые элементы встают в хвост очереди.

## 2. Архитектура системы

### 2.1. Состав модулей (4 сервиса + оркестратор)

| Модуль | Назначение | Режим |
|--------|------------|-------|
| Новости | Поиск IT-новостей, сокращение, авторский текст RU, озвучка | Cron каждые 15 мин |
| Музыка | Заполнение эфира из плейлистов Яндекс.Музыки | Автономный буфер |
| Объявления | Голосовые сообщения из Telegram с геотаргетингом | По событию |
| Telegram-бот | Объявления, заказы, плейлист, city_tag | По команде |
| Broadcast Engine | Очередь, приоритеты, HLS/WebSocket-стрим | Центральный |

### 2.2. Геотаргетинг

| Параметр | Описание |
|----------|----------|
| `city_tag` | Строковый ID города: `moscow`, `spb`, `ekb` и т.д. |
| `city_tag = "all"` | Глобальный эфир — контент слышат все слушатели |
| Определение города | 1) Geolocation API (с согласия) → reverse geocoding; 2) fallback: IP → GeoIP (MaxMind/ip-api) |
| Хранение на клиенте | `localStorage.fm21_city` + query-параметр `?city=` для ручного override |
| Изоляция эфира | Отдельная очередь контента и поток на каждый `city_tag` |

### 2.3. Поток данных эфира

1. **Music Service** наполняет базовую очередь треками из Яндекс.Музыки по правилам плейлиста.
2. **News Cron** (каждые 15 мин) добавляет слот новости с перебивкой в расписание всех городов.
3. **Ads Service** при получении голосового из Telegram вставляет объявление в очередь города отправителя.
4. **Бот** при заказе трека добавляет трек в очередь города с повышенным приоритетом (ниже объявлений, выше обычной музыки).
5. **Broadcast Engine** склеивает очередь в непрерывный поток и отдаёт клиентам через HLS или WebSocket.

## 3. Модули системы

### 3.1. Модуль «Новости»

| Аспект | Требование |
|--------|------------|
| Источник | Свободный поиск по мировым IT-новостным ресурсам (любой язык): Hacker News, TechCrunch, The Verge, Ars Technica, Habr и др. |
| Обработка текста | Сокращение до сути → авторский текст на русском (150–250 слов, ≈1–2 мин озвучки при 130 слов/мин) |
| Озвучка | API с русским языком и корректными ударениями (Нейрозвук.онлайн или аналог), `lang=ru` |
| Расписание | Cron `*/15 * * * *` — вставка слота NEWS в очередь каждого города |
| Кэш и повторы | Одна новость — не более 3 раз в сутки. Кэшировать текст и аудио, не переозвучивать при повторе |
| Перебивка | Перед новостью: `news-stinger.mp3` (3–5 сек) «Сейчас новости» |
| В эфире | Не прерывает текущий блок — вставка после завершения трека/объявления |

**Алгоритм cron-задачи новостей:**

1. Выбрать 1 новость с `play_count < 3` за 24 ч.
2. Если текст не кэширован — fetch → summarize → save.
3. Если аудио не кэшировано — TTS → upload S3 → save `audio_url`.
4. Для каждого активного `city_tag`: enqueue [{type: STINGER}, {type: NEWS, id}].
5. Инкремент `play_count`, обновить `last_played_at`.

### 3.2. Модуль «Музыка»

| Аспект | Требование |
|--------|------------|
| Источник | Яндекс.Музыка API (через server-side прокси, токен в .env) |
| Отбор | Автоматическая выборка из популярных плейлистов. Правила в `playlist-rules.js` — разработчик меняет только этот файл |
| Эфир | Музыка заполняет всё время, свободное от новостей и объявлений |
| Буфер | Фоновый воркер поддерживает ≥ 10 треков в очереди MUSIC на город |
| Заказ через бота | Резолв по названию/исполнителю → enqueue с `priority=MUSIC_ORDER` (с геотаргетингом) |
| Смена плейлиста | Команда `/playlist <name>` → обновление правил в Redis для city_tag |

### 3.3. Модуль «Объявления»

| Аспект | Требование |
|--------|------------|
| Вход | Голосовое сообщение, присланное пользователем в Telegram-бота (OGG Opus) |
| Обработка | ffmpeg: OGG → MP3 128kbps, нормализация громкости (EBU R128) |
| Геотаргетинг | Отправитель указывает город; объявление слышат только слушатели этого `city_tag` |
| Воспроизведение | Сразу после текущей песни или новости, без перебивания. `priority=AD` (максимальный) |
| Лимиты | Макс. длительность 60 сек, макс. 5 объявлений в очереди на город |

### 3.4. Модуль «Telegram-бот»

| Команда / действие | Поведение |
|--------------------|-----------|
| Голосовое сообщение | Выбор города (inline-кнопки) → сохранение объявления → enqueue в эфир города |
| `/order Название — Исполнитель` | Поиск в Яндекс.Музыке → подтверждение → enqueue для city_tag отправителя |
| `/playlist <name>` | Смена правил плейлиста (только admin) |
| `/city <tag>` | Установка city_tag для заказов и объявлений отправителя |
| `/city all` | Глобальный режим — контент для всех слушателей |
| `/status` | Текущий трек, очередь, время до следующей новости |

## 4. Очередь контента и приоритеты

### 4.1. Типы элементов очереди

| type | priority | Описание |
|------|----------|----------|
| `AD` | 100 | Голосовое объявление |
| `NEWS_STINGER` | 80 | Перебивка перед новостью (3–5 сек) |
| `NEWS` | 75 | Озвученная новость (1–2 мин) |
| `MUSIC_ORDER` | 50 | Заказанный через бота трек |
| `MUSIC` | 10 | Обычный трек из плейлиста |

### 4.2. Алгоритм Queue Manager

```
function getNextItem(cityTag):
  1. Если есть AD в pending → вернуть AD (после завершения current)
  2. Если наступил слот NEWS (scheduler) → вернуть STINGER + NEWS
  3. Если есть MUSIC_ORDER → вернуть его
  4. Иначе → вернуть MUSIC из буфера плейлиста

function enqueue(item, cityTag):
  - AD: вставить после currentIndex + 1 (не прерывает)
  - NEWS: по расписанию scheduler
  - MUSIC_ORDER: после AD, перед MUSIC
  - MUSIC: в конец буфера
```

## 6. Web-плеер (фронтенд)

### 6.1. UI-компоненты

- **Логотип FM21** — градиент #44EB99 → #861BE3

### 6.3. Определение города при старте

```
async function detectCity():
  1. Проверить ?city= в URL и localStorage
  2. navigator.geolocation.getCurrentPosition (timeout 5s)
     → GET /api/geo/reverse?lat=&lon=
  3. Fallback — GET /api/geo/detect (IP)
  4. Сохранить в localStorage, показать бейдж
  5. Инициализировать stream-client с cityTag
```

### 6.4. Непрерывное воспроизведение и фон

- AudioContext + GainNode для громкости пользователя.
- Crossfade 2 сек между треками; prefetch следующего сегмента за 10 сек.
- Media Session API — метаданные на lock screen.
- При `document.hidden` — не останавливать контекст; reconnect WS при обрыве.

### 7.1. Публичные эндпоинты

## 12. Критерии приёмки

| Область | Критерий |
|---------|----------|
| Геотаргетинг | Город определяется при старте; объявления и заказы изолированы по city_tag |
| Веб-клиент | Эфир без авторизации; громкость 0–100%; цвета #44EB99 и #861BE3 |
| Фон | Звук продолжается при сворачивании вкладки (Chrome/Firefox/Safari) |
| Новости | Каждые 15 мин; 1–2 мин; повтор ≤ 3/сутки без переозвучки; перебивка 3–5 сек |
| Музыка | Заполняет эфир; playlist-rules.js меняется без правок ядра |
| Объявления | Voice из Telegram после трека/новости, только для своего города |
| Бот | Голосовые, /order, /playlist, /city — с подтверждением |
| Очередь | Приоритет: объявления > новости > музыка; без перебивания |
| Интеграции | Yandex Music proxy, TTS, Telegram webhook на сервере |

## 13. Риски и митигация

| Риск | Митигация |
|------|-----------|
| Нестабильность API Яндекс.Музыки | Абстракция MusicProvider, fallback на локальную библиотеку |
| Autoplay policy в браузерах | Обязательная кнопка «Слушать» |
| Неточность GeoIP | Ручной выбор города в UI + override `?city=` |
| Задержка TTS при cron | Предгенерация новости за 2 мин до слота |
| Лицензирование музыки | Только легальные stream URL из подписки |
